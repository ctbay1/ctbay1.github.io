The NLP Project Store: E-commerce Chatbot Demo with Flask and NLP (Python)

1. Introduction

1.1. Overview

Project link: https://ctbay.pythonanywhere.com

Source code: Click here

This project showcases the development of an e-commerce chatbot designed to enhance customer service interactions within an online clothing store. The chatbot aims to understand natural language queries, identify customer intents, extract relevant entities, and provide accurate responses. By leveraging Natural Language Processing (NLP) techniques, including intent classification and named entity recognition (NER), this chatbot provides a streamlined and informative user experience.

The project goes beyond the chatbot itself, incorporating a basic online store frontend built with Flask, HTML, CSS, and JavaScript. This interactive interface demonstrates the chatbot's capabilities in a realistic setting, allowing users to browse products, add items to their cart, and interact with the chatbot for various inquiries.

1.2. Goals

The primary goal of this project is to demonstrate and solidify my knowledge of NLP and machine learning concepts acquired through Codecademy's course. By creating a practical, functional chatbot, I aimed to showcase my ability to:

  • Accurately classify user intents with an accuracy target of at least 90%.
  • Extract relevant entities (products, sizes, colors, etc.) from natural language queries.
  • Provide informative and helpful responses based on user queries.
  • Integrate the chatbot into a web application to provide a seamless user experience.

1.3. Motivation

My interest in NLP and AI, fueled by their growing importance in modern technology, inspired this project. Recognizing the widespread use of chatbots in customer service, I saw an opportunity to explore this exciting field while creating a practical application that could potentially improve the online shopping experience.

This project served as a challenging yet rewarding exploration of NLP and web development, demonstrating the potential of AI to automate and enhance customer interactions in the e-commerce domain.

3. Natural Language Processing (NLP) Components

The e-commerce chatbot leverages natural language processing (NLP) to understand user queries and provide relevant responses. The core NLP components consist of:

3.1. Intent Classification

3.1.1. Approach

A Naive Bayes classifier is employed to determine the intent of a user's message. This probabilistic model calculates the probability of a message belonging to each predefined intent based on the words it contains. The intent with the highest probability is chosen as the most likely one.

3.1.2. Dataset Creation and Refinement

To train the classifier, an intent_data.csv dataset was meticulously curated, containing diverse user utterances paired with their corresponding intents. Chatbots and other resources were utilized to automatically generate a wider variety of sentence structures. Given the limited resources of an individual project, the dataset is not exhaustive, but it was designed to be representative of common user queries.

This dataset underwent several iterations of refinement:

  • Brainstorming: A variety of potential user queries were brainstormed for each intent, focusing on capturing diverse phrasing, vocabulary, and sentence structures.
  • Data Augmentation: Additional queries were generated using chatbot tools to increase the dataset's diversity.
  • Labeling: Each utterance was manually labeled with its correct intent. This involved carefully assigning each utterance to one of the predefined categories (e.g., order_status, returns, stock_info, etc.).
  • Error Analysis and Refinement The classifier's performance was evaluated, and the resulting confusion matrix was analyzed. Misclassified utterances were identified, and the dataset was further refined by adding more variations and emphasizing intent-specific keywords. This iterative process significantly improved the model's accuracy from an initial score below 0.5 to a final score of 0.95.

3.1.3. Model Training and Evaluation

The Naive Bayes classifier was trained on the refined dataset using Scikit-learn. The model's performance was evaluated using a variety of metrics, such as its accuracy score, and confusion matrix. The final model achieved an accuracy of 95%, indicating its ability to accurately classify user intents.

3.1.4. Adressing Challenges

The limited dataset size posed a significant challenge, as it constrained the model's ability to learn subtle nuances in user language. However, by carefully crafting and enriching the dataset with intent-specific keywords, the accuracy was significantly improved.

Ambiguity between certain intents (e.g., "shipping_info" vs. "order_status") remains a challenge, especially in cases where the user's query lacks specific context. Future enhancements could involve gathering additional data to improve the model's ability to disambiguate these cases or incorporating contextual information into the classification process.

3.2. Entity Extraction

To identify specific items, their sizes, colors, and order numbers mentioned in user queries, a hybrid approach combining Named Entity Recognition (NER) with a rule-based color identification system was implemented.

3.2.1. Named Entity Recognition (NER) with SpaCy

spaCy NER Model: The core of the entity extraction relies on a custom spaCy NER model trained to recognize product types (PRODUCT) and sizes (SIZE). The model was trained on a dataset of diverse user queries, meticulously labeled using the Label Studio annotation tool.

A second NER model was also trained to recognize the combined PRODUCT_DETAILS entity, which encompasses the product name, color, and size information. This model addresses the challenge of associating these entities when multiple products are mentioned in a single query.

Model 1:Trained to identify individual entities (PRODUCT, SIZE) in user queries.

  • PRODUCT: General clothing types (e.g., "shirt", "jeans", "dress").
  • SIZE: Adjective and numeric sizes (e.g., "small", "large", "waist size 34", "32x32")

Model 2: Trained to recognize combined entities (PRODUCT_DETAILS) for more complex queries.

  • PRODUCT_DETAILS: Captures the full product description, including product name, color, and size (e.g., "blue jeans in a size 32").

This two-model strategy was crucial to address the challenge of accurately associating sizes and colors with their corresponding products, especially in cases where multiple items were mentioned in a single query.

3.2.2. Data Labeling

An initial set of sentences was generated using the utterances previously crafted for intent classification. To enhance diversity, additional sentences were created using various techniques: including brainstorming diverse product names, sizes, colors, and their combinations and utilizing chatbots for generating diverse sentences.

A dataset of approximately 100 sentences was manually labeled using Label Studio, an open-source data annotation tool.

Labeling Guidelines:

  • PRODUCT: Only the general clothing type (e.g., “shirt,” “dress”) was labeled, excluding specific details like style or material.
  • SIZE: The entire size phrase was labeled, including both adjective sizes (e.g., “small”) and numerical sizes (e.g., “waist size 34”).
  • PRODUCT_DETAILS: The full product description, including the product name, color, and size (e.g., "blue jeans in a size 32").
  • COLOR: Initially considered for NER, but a rule-based approach was ultimately adopted using a comprehensive list of colors derived from a Wikipedia resource: https://github.com/k-kawakami/colorfulnet/blob/master/example_data/wikipedia-list-of-colors.txt. A straightforward rule-based approach was implemented to identify colors mentioned in the user's message. This involved checking if any of the words in the message match the colors in the predefined list. This is a practical approach since color names are relatively well-defined and easier to identify with a straightforward lookup.

After the labeling was done, the resulting JSON data was loaded into a list.

  • JSON Data: The labeled data was stored in JSON format.
  • Data Loading: The JSON data was loaded into a list for further processing.

3.2.3. Model Training

Both spaCy NER models were trained using the labeled dataset. The training process involved:

  1. Initialization: The spaCy model was initialized with a blank English model.
  2. Adding labels: The labels (PRODUCT, SIZE, PRODUCT_DETAILS) were added to the respective NER model.
  3. Training loop: The model was trained using the labeled dataset for a specified number of iterations.

3.2.4. Evaluation and Adressing Challenges

Evaluation:

  • Non-continuous product descriptions: One challenge encountered during entity extraction was the ambiguity arising from non-continuous product descriptions (e.g., "I like those blue jeans... are they available in a size 30?").
  • Ambiguous queries: Queries that mentioned multiple products or sizes in a single sentence posed a challenge for the NER model.

Addressing Challenges: To enhance entity recognition, the following strategies were employed:

  • Refined Training Data: The dataset was enriched with more negative examples to reduce overfitting to specific patterns. Additionally, wider range of variations for size formats were included to improve the model's ability to recognize diverse size representations.
  • Iterative training: Multiple iterations of training were conducted, varying the number of training epochs to find the optimal balance between underfitting and overfitting. It was discovered that around 50 iterations resulted in the best performance, likely due to the limitations of the small dataset.
  • Ambiguous Product Descriptions: Some product descriptions were non-continuous or contained ambiguous terms. This was addressed by labeling them as a single PRODUCT_DETAILS entity and then using a separate NER model trained on complete product descriptions to extract the individual components (product, size, color).

4. Chatbot Implementation

This section delves into the core logic of the chatbot, focusing on how it processes user input, classifies intent, extracts relevant entities, and generates appropriate responses.

4.1. Chatbot Architecture

The chatbot is designed as a hybrid system that combines aspects of both rule-based and retrieval-based architectures.

  • Rule-based: It incorporates rule-based logic to handle specific scenarios and tasks. This is evident in the directing_function, which determines the flow of the conversation and calls appropriate functions based on the identified intent (e.g., retrieving order status, adding items to the cart, or providing product information).
  • Retrieval-based: It primarily operates by retrieving pre-defined responses from the responses.csv file. The intent classification model helps select the most appropriate response based on the user's query.

4.2. Intent Classification

The chatbot's find_intent function is responsible for classifying the user's intent based on their message. The chatbot employs a Naive Bayes classifier to determine the intent behind a user's message. The classifier has been trained on a carefully curated dataset (intent_data.csv) that encompasses a wide range of user queries and their associated intents (e.g., order_status, returns, stock_info).

To enhance the chatbot's ability to handle irrelevant or out-of-scope queries, a probability threshold of 0.2 is employed. After the Naive Bayes classifier predicts the intent, the model's confidence in that prediction is assessed. If the probability of the predicted intent falls below the threshold, the chatbot defaults to a "I'm sorry, I didn't quite understand that. Could you rephrase your question?" response, prompting the user to try again.

4.3. Entity Extraction

The chatbot employs two distinct spaCy NER models to extract relevant entities from user messages:

  • Model 1 (nlp_label_productName_size): Extracts individual entities such as PRODUCT (e.g., "shirt", "jeans") and SIZE (e.g., "small", "waist size 30"). This model is used for both single-product and multiple-product queries to identify basic product information.
  • Model 2 (nlp_label_product_details): Identifies entire product descriptions as PRODUCT_DETAILS entities. This model is specifically designed to address the ambiguity of multiple products mentions by grouping product names, sizes, and colors together.

The find_entities function utilizes these two models strategically to extract information, prioritizing Model 2 for queries with multiple products mentioned to resolve potential ambiguities in entity association. The get_size_product_color_from_extracted_entities function further processes the extracted entities, cleaning and organizing them into a standardized format.

4.4. Managing Conversational State and Context

The chatbot maintains a minimal level of conversational state using a self.state dictionary. This state is primarily used to track situations where the chatbot requires additional input from the user to complete a specific task.

Example: Order Status Inquiry

For example, when the user's intent is identified as order_status, but their order number isn't explicitly mentioned in their message, the chatbot:

  1. Sets self.state['current_task'] to "waiting for an input".
  2. Sets self.state['intent_label'] to order_status.
  3. Prompts the user to provide their order number.

In subsequent interactions, the chatbot checks the self.state to determine if it's waiting for an input. If so, it processes the user's input (e.g., the order number) to fulfill the previous request.

This allows the chatbot to "remember" that it's waiting for an order number in the next user message. When the user provides the number, the chatbot can then proceed with the order status lookup.

4.5. Chatbot Response Generation Summary

The overall interaction flow of the chatbot can be summarized as follows:

  1. User input: The chatbot.respond function is called in the Flask routes, passing the user's message as input.
  2. Intent Classification: The find_intent function identifies the user's intent (e.g., order_status, add_cart) using a Naive Bayes classifier trained on the intent_data.csv dataset.
  3. Entity Extraction (Optional): If the intent requires it (e.g., stock_info, add_cart), the find_entities function extracts relevant entities (products, sizes, colors) from the user's message. If multiple products are mentioned, the second NER model (nlp_label_product_details) is used to group entities.
  4. Response Generation:
    • If the chatbot is waiting for additional input (as indicated by self.state), it processes the user's input and generates a response accordingly.
    • If the chatbot does not require further input, it generates a response based on the identified intent and entities.
    The directing_function method handles the intent-specific logic, possibly updating the self.state if further input is needed, and generates an appropriate response.
  5. Response Display: The response is returned to the frontend, where it is displayed to the user in the chat interface.

Key points:

  • The respond function is the main entry point for handling user messages.
  • find_intent and find_entities are the core NLP functions.
  • The directing_function manages the flow of the conversation based on the identified intent.
  • self.state is used to manage limited conversational context for scenarios where the chatbot needs additional input.

5. Web Application Development (Flask)

5.1. Overview

The chatbot is integrated into a Flask (a lightweight Python web framework) web application designed to simulate a basic online clothing store. This serves as a practical demonstration of the chatbot's capabilities in a realistic e-commerce setting. The application allows users to:

  • Browse product listings.
  • View individual product details.
  • Add items to their cart.
  • View their cart contents.
  • Log in to a simulated user profile.
  • Interact with the chatbot throughout the website.

Flask handles routing, template rendering, and the backend logic for user interactions, product display, and chatbot integration.

5.2. Flask Routes and Templates

The Flask application defines several routes to handle different pages and actions:

  • /(index): The homepage displays a list of available products fetched from the inventory.csv file.
  • /product/<product_type> (categories): Displays a filtered list of products based on their category (e.g., shirts, pants).
  • /product/<product_type>/<productSize_and_color> (product page) Shows details for a specific product, including an "Add to Cart" button.
  • /cart Renders the user's shopping cart, allowing them to view and modify its contents.
  • /login Handles user login with authentication using a CSV file containing mock user data.
  • /profile/<username> Displays the user's profile page, showing their order history and personal information.

Each route is associated with a corresponding HTML template (e.g., index.html, product_page.html, cart_page.html, etc.) that is rendered to present the information to the user.

Homepage (/) and Category Pages (/product/<product_type>): The homepage and category pages share a similar structure. They display a list of products fetched from the inventory.csv file. Each product is presented with an image (or placeholder), name, size, color, and price. Users can click on a product to view its details on the product page.

Product Page (/product/<product_type>/<productSize_and_color>): The product page provides detailed information about a specific product. It fetches the product details from the inventory.csv file based on the product_type and productSize_and_color parameters passed in the URL. The page displays the product's image, name, description, price, available sizes and colors, and stock quantity. The "Add to Cart" button is also present on this page.

Shopping Cart (/cart): The cart page displays a list of items that the user has added to their cart. Each item is displayed with its name, size, color, quantity, and price. The cart calculates the total price based on the items and their quantities.

User Login (/login) and Profile Page (/profile/<username>): The login page provides a form for users to enter their username and password. It then authenticates the user against a list of registered users stored in the sample_users.csv file using werkzeug.security for password hashing. If the login is successful, the user is redirected to their profile page (/profile/<username>). The profile page displays user-specific information (e.g., the latest order, order history).

5.3. JavaScript Integration

JavaScript is used for dynamic interactions on the website, including:

  • Fetch API: It is used to send user messages to the Flask backend and receive the chatbot's responses asynchronously, avoiding full page refreshes and creating a smooth, chat-like interaction.
  • Category Selection: JavaScript is used to toggle the visibility of the categories section on the homepage.

5.4. Flask Backend and Sessions

Flask handles the backend logic, including:

  • Routing: Managing different URL paths and associated actions.
  • Rendering templates: Generating HTML pages based on templates and data.
  • Chatbot Interaction: Receiving user messages, processing them with the chatbot, and returning appropriate responses.
  • Session management: Using sessions to store user-specific data, like the chatbot object, shopping cart, and login status.

Flask sessions: Flask sessions are used to maintain user-specific data across multiple requests. This includes the user's login status, cart information, and the state of the chatbot's conversation.

The ChatBot object is stored in the session and accessed in all relevant routes to ensure a seamless chatbot experience for each user.

6. Challenges and Learnings

Building the NLP Project Store chatbot presented several challenges, primarily stemming from the limitations of working with a small dataset, the inherent ambiguity of natural language, and the intricacies of deploying a web application with integrated AI components.
However, these challenges also provided valuable learning opportunities and insights about how to address them effectively.

6.1. Intent Classification Challenges

The limited size of the intent_data.csv dataset posed a significant hurdle. With only a few examples per intent initially, the Naive Bayes classifier struggled to generalize and accurately classify user queries, leading to an initial accuracy score of less than 50%.
To address this, an iterative process of data augmentation and refinement was employed:

  • Data Augmentation: Synonyms, paraphrasing, and sentence restructuring techniques were used to expand the dataset, introducing variations in language while preserving the core intent.
  • Keyword Emphasis: Intent-specific keywords were identified and emphasized in the dataset to help the model learn the distinguishing features of each intent.
  • Error Analysis: Thorough analysis of the confusion matrix and misclassified examples guided the refinement process, highlighting areas where the classifier needed more examples or clarity.

Through these efforts, the accuracy of the intent classifier was significantly improved, ultimately reaching an average score of 95.4% across multiple evaluations. This demonstrates the effectiveness of a systematic approach to dataset enhancement, even when working with limited resources.

6.2. Entity Extraction Challenges

The limited size of the dataset posed a challenge, as it constrained the model's ability to learn subtle nuances in user language and generalize to a wider range of product descriptions.
To mitigate this, an iterative approach to dataset refinement was adopted:

  • Two-model Approach: A two-model strategy was adopted to overcome the ambiguity of multiple products. One model focuses on extracting individual entities (PRODUCT, SIZE), while the other identifies entire product descriptions (PRODUCT_DETAILS), consolidating product names, colors, and sizes. This addresses ambiguity when multiple products are mentioned within a single query.
  • Rule-based Color identification: A comprehensive color list was used to efficiently identify COLOR entities, bypassing the need for extensive training data.
  • Negative Examples: Sentences were added to the dataset where words or phrases commonly appearing in product descriptions were used in different contexts. For instance, the model initially misidentified the word "some" in the sentence "Can you show me some blue colored jeans?" as a PRODUCT entity. To address this, more examples containing "some" in non-product contexts were added to the dataset to help the model better distinguish its meaning.
  • Expanding Vocabulary: To improve the model's ability to recognize a variety of clothing items, sizes, and styles, the dataset was expanded with numerous examples encompassing a broader range of terms. This included variations like "women's running shoes," "32x34 jeans," and "black leather jacket," ensuring that the model could handle the diverse expressions users might employ.

Observation
The number of iterations during NER model training didn't directly correlate with improved entity extraction performance. Experiments with 20, 50, and 100 iterations revealed that 50 iterations yielded the best results, suggesting that the small dataset size might lead to overfitting with higher iteration counts.

6.3. Persisting Chatbot State (Pickle)

Integrating the chatbot object within a Flask session posed an additional challenge. The complex nature of the ChatBot class prevented it from being directly stored in the session due to serialization limitations.
To overcome this, the pickle module was utilized to serialize and deserialize the chatbot object, allowing its state and attributes to be preserved across user sessions.

6.4. Deployment Challenges

The deployment process on PythonAnywhere presented a few hurdles:

  • File Structure: Ensuring the correct file structure and paths for the Flask application was crucial for proper deployment.
  • Environment Setup (NLTK Data): Installing the necessary packages on PythonAnywhere required careful attention to dependencies. The requirement to manually download and configure the NLTK data (stopwords, punkt, wordnet) was an additional step in the deployment process.
  • Virtual Environment Setup: Ensuring that all Python packages, including spaCy, were correctly installed and accessible within the virtual environment was essential for the chatbot to function properly.

By carefully addressing these challenges, the application was successfully deployed and made accessible online.

6.5. Learnings and Takeaways

The NLP Project Store provided valuable insights and learnings in various aspects of natural language processing, chatbot development, and web application deployment:

  • Data Augmentation: The importance of data augmentation in enhancing model performance and generalization. Even with a small dataset, careful curation and iterative refinement can lead to significant improvements in model performance. A larger and more diverse dataset would likely lead to further accuracy gains.
  • Iterative Training: The benefits of iterative training and dataset refinement to address model limitations and improve accuracy.
  • Hybrid Architectures: The advantages of combining rule-based and retrieval-based approaches in chatbot design to handle diverse user queries effectively.
  • Session Management: The significance of session management in web applications to maintain user-specific data and state.
  • Deployment Challenges: The complexities involved in deploying AI-powered applications and the importance of meticulous setup and configuration.

Overall, the NLP Project Store served as a comprehensive learning experience, highlighting the intricacies and challenges of developing AI-powered applications while providing practical insights into effective strategies for overcoming these challenges.

With real-world testing, the model can be further refined and expanded to handle a broader range of user queries and product descriptions.
